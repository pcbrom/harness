### Phase 1 (base architecture, 4 harnesses, claude adapter)

Added:

- Package skeleton with a valid DESCRIPTION, an MIT license and a NAMESPACE
  generated by roxygen2.
- Public API: `status()`, `setup()`, `available_roles()`, `role()` and
  `launch()`. Exported helpers: `scaffold_layout()`, `community_skills_path()`
  and `adapters()`.
- Harness loader and validator in `R/harness.R`, reading
  `inst/harness/<role>.yml` and checking the schema. The manual execution
  policy is structural: a harness that does not declare
  `execution_policy: manual` is rejected at load time.
- Four curated harnesses: `data-scientist`, `statistician`,
  `package-maintainer` and `paper-author`.
- Reference `claude` adapter in `R/adapter_claude.R`: binary discovery,
  non-destructive merge into `settings.json`, writing the role system prompt
  into the project `.claude` directory, and symlinking the curated SKILL.md
  files present in the community-skills checkout.
- Discovery of community-skills through `COMMUNITY_SKILLS_PATH`,
  `~/.community-skills/` or `~/projects/community-skills/`, without bundling the
  catalogue in the tarball.
- `clone_community_skills()` to fetch the external catalogue into a discoverable
  path, without bundling it. The startup message points at the command when the
  catalogue is not found, without accessing the network on load.
- `update_community_skills()` for `git pull --ff-only` on the checkout.
  Load-time auto-update is opt-in through the `harness.auto_update` option or
  the `HARNESS_AUTO_UPDATE` environment variable; by default the package does
  not access the network on load.
- Launch through `rstudioapi::terminalCreate` when inside RStudio, with a
  fallback to an external terminal emulator and, when none is available, to
  reporting the command for the user to run. Each launch opens a terminal tab
  with a unique caption, so repeated launches do not collide.
- `role_list()`: a table of the roles with version, skill count and
  description.
- `opencode` and `codex` adapters, pulled forward from later phases: generation
  of `opencode.json` and `.codex/config.json` by non-destructive merge, the
  role prompt in `AGENTS.md` (the cross-tool convention, without overwriting an
  existing file) and symlinking of the curated SKILL.md files. Shared adapter
  helpers extracted for common use (`harness_link_skills`,
  `harness_config_block`, `harness_write_agents`, `harness_write_json_config`).
- testthat suite with smoke tests for the four harnesses and the adapters, with
  a mocked binary and a fixture checkout.
